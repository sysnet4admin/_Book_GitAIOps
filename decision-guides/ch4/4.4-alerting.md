# 4.4 알림 방식 선택

## 상황

4.2에서 Prometheus와 Grafana를 설치하고, 4.3에서 Loki로 로그를 수집하고 있다. 메트릭과 로그를 볼 수는 있지만, 대시보드를 24시간 지켜볼 수는 없다. 문제가 생기면 자동으로 알려주는 알림 시스템이 필요하다.

## 독자가 물어볼 수 있는 질문

- "알림은 어떻게 설정해?"
- "Prometheus 알림이랑 Grafana 알림이 뭐가 달라?"
- "Slack이나 PagerDuty로 알림 보낼 수 있어?"
- "어떤 조건에서 알림을 보내야 해?"

## 추천: PrometheusRule + Alertmanager

kube-prometheus-stack에 이미 포함된 Prometheus Operator의 `PrometheusRule` CRD로 Alert Rule을 YAML로 작성하고, 같이 설치된 Alertmanager가 알림을 라우팅한다. 규칙이 코드로 남고 ArgoCD가 동기화하므로 GitOps 흐름과 자연스럽게 맞물린다.

이 프로젝트에 적합한 이유:
- **GitOps 호환**: `PrometheusRule` CRD를 매니페스트로 관리한다. Git 저장소에 규칙이 누적되고 ArgoCD가 클러스터에 적용한다
- **이미 설치됨**: 4.2에서 kube-prometheus-stack을 설치하면 Alertmanager + PrometheusRule CRD가 함께 들어온다 → 추가 설치 불필요
- **표준 도구**: 실무에서 가장 보편적인 알림 스택. 책 전반의 GitOps 우선 원칙과 일치
- **강력한 라우팅**: Alertmanager의 그루핑/억제/라우팅 트리가 다단계 알림을 표현한다

## 비교

| 방식 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **PrometheusRule + Alertmanager** | Prometheus 스택 네이티브 | GitOps 호환(CRD), 강력한 라우팅/그루핑, 실무 표준 | YAML 작성 필요, Alertmanager 라우팅 학습 | ★★★ |
| **Grafana Alerting** | Grafana 내장 알림 | UI로 빠르게 시작, 다양한 채널 | UI 기반이라 GitOps와 어긋남, 대규모에서 Alertmanager보다 약함 | ★★ |
| **PagerDuty / OpsGenie** | 전문 인시던트 관리 | 에스컬레이션, 온콜 스케줄, 모바일 앱 | 유료, 학습 환경에서 과도 | ★ |
| **Cloud Monitoring Alert** | GCP 네이티브 | GCP 콘솔 통합, SMS 지원 | Prometheus 메트릭 직접 못 씀, 별도 설정 | ★ |

> **Notiflex 맥락**: 책 전반에서 GitOps(YAML → Git → ArgoCD) 우선 원칙을 따랐다. 알림도 같은 흐름으로 들어가야 일관성이 깨지지 않는다. Grafana UI 알림은 빠르게 시작할 수 있지만 규칙이 클러스터 안에 머물러 Git 저장소에 남지 않는다. PrometheusRule + Alertmanager는 첫 설정에 약간 더 손이 가지만 한 번 패턴이 잡히면 다음 알림은 YAML 한 파일 추가로 끝난다.

## 핵심 개념

- **Alert Rule**: "이 조건이 N분 동안 지속되면 알림을 발생시켜라"는 규칙. 예: `kube_pod_container_status_restarts_total > 0`이 5분간 지속.
- **Contact Point**: 알림을 받을 대상. Slack webhook, 이메일, PagerDuty 등을 설정한다.
- **Evaluation Interval**: Alert Rule을 얼마나 자주 평가할지. 1분이면 매분 조건을 확인한다.
- **Notification Policy**: 어떤 알림을 어떤 Contact Point로 보낼지 라우팅 규칙. 심각도별로 다른 채널에 보낼 수 있다.
