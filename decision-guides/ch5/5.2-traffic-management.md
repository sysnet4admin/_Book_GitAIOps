# 5.2 외부 트래픽 관리 방식 선택

## 상황

Notiflex API는 ClusterIP Service로 클러스터 내부에서만 접근 가능하다. 외부에서 HTTP 요청을 받으려면 트래픽을 클러스터 안으로 라우팅하는 방법이 필요하다.

## 독자가 물어볼 수 있는 질문

- "외부에서 API에 접근하려면 어떻게 해?"
- "Ingress가 뭐야?"
- "Gateway API가 뭐야? Ingress랑 뭐가 달라?"
- "외부 트래픽 관리 도구 추천해줘"

## 추천: Gateway API

Gateway API는 Kubernetes의 차세대 트래픽 관리 표준이다. 기존 Ingress 리소스의 한계를 극복하기 위해 설계되었고, GKE에서 네이티브로 지원한다.

이 프로젝트에 적합한 이유:
- **K8s 공식 표준**: Ingress를 대체하는 차세대 API (GA since K8s 1.27)
- **GKE 네이티브**: 별도 Ingress Controller 설치 없이 GKE가 자동으로 처리
- **역할 분리**: Gateway(인프라팀) / HTTPRoute(앱팀)로 관심사 분리
- **5.3 Blue/Green 연동**: HTTPRoute의 backendRefs로 트래픽 분배 가능

## 비교

| 도구 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **Gateway API** | K8s 표준, 역할 분리 | GKE 네이티브, 설치 불필요, 확장성 | 비교적 새로움 | ★★★ |
| **Ingress NGINX** | 가장 보편적 | 사용례 풍부, 커뮤니티 활발 | 별도 Controller 설치 필요, 단일 리소스에 모든 설정 | ★★ |
| **Istio** | 서비스 메시, 고급 트래픽 관리 | mTLS, 카나리, 서킷브레이커 | 매우 무거움(1GB+), 복잡 | ★ |
| **Traefik** | 자동 서비스 발견, Let's Encrypt 통합 | 설정 간편, 동적 라우팅 | NGINX 대비 생태계 작음 | ★★ |

> **Notiflex 맥락**: GKE Standard에서 Gateway API는 추가 설치 없이 바로 사용할 수 있다. GKE의 GatewayClass `gke-l7-regional-external-managed`가 자동으로 로드밸런서를 생성한다. Istio는 리소스가 부족하고, NGINX는 별도 설치가 필요하다.

## 핵심 개념

- **Gateway**: "어떤 IP/포트로 트래픽을 받을 것인가"를 정의한다. 인프라팀이 관리한다.
- **HTTPRoute**: "어떤 경로의 요청을 어떤 Service로 보낼 것인가"를 정의한다. 앱팀이 관리한다.
- **GatewayClass**: Gateway의 구현체를 지정한다. GKE는 `gke-l7-regional-external-managed`(리전 외부 HTTP), `gke-l7-global-external-managed`(글로벌 외부 HTTP), `gke-l7-rilb`(내부) 등을 제공한다. Notiflex는 단일 리전(asia-northeast3)에서 운영하므로 `gke-l7-regional-external-managed`를 사용한다.
- **HealthCheckPolicy**: GKE 전용 CRD. Gateway가 백엔드 Pod의 상태를 확인하는 방법을 정의한다.
