# 7.3 App of Apps 패턴

## 실행 지침

### 단계 1: App of Apps 구조 생성

`argocd/root-app.yaml` — 루트 Application:
```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: root-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/<USER>/notiflex-platform.git
    targetRevision: main
    path: argocd/apps
    directory:
      recurse: true
  destination:
    server: https://kubernetes.default.svc
    namespace: argocd
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

`argocd/apps/` 디렉터리에 각 Application 매니페스트를 배치한다. 루트 Application이 이 디렉터리를 감시하여 하위 Application들을 자동 관리한다. `argocd/` 직속에는 `root-app.yaml`만 둔다(루트가 자기 자신을 다시 sync 대상으로 잡는 순환을 피하기 위함).

> ⚠️ **directory.recurse 필수**: `argocd/apps/` 하위에 `smb/`, `enterprise/` 등 서브디렉터리를 사용하면 `directory.recurse: true`를 설정해야 한다. 이 옵션이 없으면 ArgoCD가 루트 디렉터리의 YAML만 읽고 서브디렉터리의 Application을 생성하지 않는다.

### 단계 2: 적용

```bash
kubectl --context gke-sysnet4admin_book_gitaiops apply -f argocd/root-app.yaml
```

### 단계 3: Sync Wave로 설치 순서 지정

App of Apps가 동작하면 `argocd/apps/` 안의 Application들이 한꺼번에 sync된다. 인프라(Gateway API, Namespace) → 플랫폼(monitoring) → 애플리케이션(notiflex-smb) 순서로 설치되어야 의존성이 깨지지 않으므로, 각 Application 메타데이터에 `argocd.argoproj.io/sync-wave` 어노테이션을 추가한다. 숫자가 낮은 wave부터 차례로 sync한다.

```yaml
# argocd/apps/monitoring.yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "1"   # 0: 인프라(Gateway/Namespace), 1: 플랫폼, 2: 앱
```

```yaml
# argocd/apps/notiflex-smb.yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "2"
```

> 💡 **sync-wave 권장 범위**: 0(인프라) / 1(플랫폼: 모니터링·Kafka 등) / 2(애플리케이션) 3단계로 충분하다. wave 간 차이는 sync 순서만 결정하고 대기 시간은 부여하지 않는다 — 앞 wave가 Healthy가 되어야 다음 wave가 시작된다.
