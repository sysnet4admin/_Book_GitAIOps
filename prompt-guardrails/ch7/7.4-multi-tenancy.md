# 7.4 멀티테넌시

## 사전 조건
- 멀티 노드풀 구성 완료 (ch7.2) — api-pool에 enterprise 앱도 배치한다.
- App of Apps 패턴 적용 완료 (ch7.3) — enterprise Application을 root-app이 관리한다.

## 실행 지침

### 단계 1: 테넌트별 Namespace 분리

enterprise 테넌트를 위한 매니페스트를 `k8s/enterprise/` 디렉터리에 생성한다:
- `namespace.yaml`: enterprise 네임스페이스
- `rollout.yaml`: Notiflex Rollout (enterprise 네임스페이스)
- `service.yaml`: Service
- `valkey-secret.yaml`: Valkey 인증 정보

> ⚠️ **cross-namespace Valkey 접근**: Enterprise 테넌트의 앱이 Valkey에 연결하려면, Rollout에 `VALKEY_ADDR` 환경변수를 cross-namespace DNS로 설정해야 한다: `valkey-primary.notiflex.svc.cluster.local:6379`. 이 설정이 없으면 앱이 `localhost:6379`로 접속을 시도하여 연결 실패 후 Pod이 Error/CrashLoopBackOff 상태가 된다. Valkey Secret도 enterprise 네임스페이스에 별도로 생성해야 한다.

> ⚠️ **Valkey 패스워드 일치 필수**: enterprise의 `valkey-secret.yaml`에 넣는 패스워드는 **반드시 실제 Valkey 인스턴스의 패스워드와 동일**해야 한다. Helm으로 설치한 Valkey의 패스워드는 `kubectl get secret valkey -n notiflex -o jsonpath='{.data.valkey-password}' | base64 -d`로 확인한다. 하드코딩된 임의의 값을 사용하면 `WRONGPASS` 에러로 Pod이 Error 상태에 빠진다.

### 단계 2: ArgoCD Application 추가

`argocd/apps/notiflex-enterprise.yaml`을 생성하여 `k8s/enterprise/` 디렉터리를 감시하도록 한다. 7.3에서 만든 root-app이 이 파일을 자동으로 감지해 enterprise Application을 생성한다.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: notiflex-enterprise
  namespace: argocd
  annotations:
    argocd.argoproj.io/sync-wave: "2"
spec:
  project: default
  source:
    repoURL: https://github.com/<USER>/notiflex-platform.git
    targetRevision: main
    path: k8s/enterprise
  destination:
    server: https://kubernetes.default.svc
    namespace: enterprise
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
    - CreateNamespace=true
```

> 💡 **`CreateNamespace=true` 필수**: enterprise Namespace를 별도 매니페스트로 분리하지 않고 ArgoCD가 sync 시점에 자동 생성하도록 위임한다. 이 옵션이 없으면 sync가 "namespace not found"로 실패하거나 독자가 사전에 `kubectl create namespace enterprise`를 직접 실행해야 한다.
