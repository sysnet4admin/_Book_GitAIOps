# 9.2 쌓인 것들을 돌아보기

## 실행 지침

### 단계 1: CLAUDE.md 성장 분석

notiflex-platform의 CLAUDE.md가 어떻게 성장해왔는지 분석한다:
- 초기 내용 vs 최종 내용
- 추가된 컨텍스트와 규칙

### 단계 2: 도구 선택 의사결정 종합

`JOURNEY.md`의 **도구 선택 기록** 테이블을 읽어서 독자가 실제로 선택한 도구들을 종합한다.

각 선택에 대해:
- 독자가 선택한 도구와 이유
- 검토했던 대안
- 트레이드오프

> 📌 **핵심**: `JOURNEY.md`에 축적된 의사결정 기록이 여기서 종합된다. 대화가 압축되어도 파일 기반으로 정확한 회고가 가능하다. 이것이 GitAIOps에서 "살아있는 의사결정 기록"이다.

### 단계 3: 일관된 선택 기준 패턴 찾기

여러 의사결정에서 반복적으로 나타나는 선택 기준을 추출한다. 책 전반에서 다음 4가지 패턴이 반복된다 — 회고에서 이를 명시적으로 짚어낸다.

1. **GKE/managed 우선** (ch3 ArgoCD 설치, ch5 GatewayClass 리전 선택, ch6 Secret Manager + GKE managed CSI): 학습 환경에서 운영 부담을 줄이고 클라우드 네이티브를 익히는 선택
2. **Argo 생태계 통일** (ArgoCD + Argo Rollouts): 단일 벤더의 일관된 UX와 통합된 도구 체인
3. **Grafana 통합 관측** (Prometheus + Loki + Tempo 모두 Grafana 데이터소스): 한 화면에서 메트릭·로그·트레이스 상관 분석
4. **GitOps 호환** (PrometheusRule, ADR, manifest YAML 모두 Git에 누적): UI 설정보다 코드 기반 관리 우선

이 4가지 패턴이 어떻게 책 진행을 일관되게 만들었는지 산문으로 짧게 회고한다.

### 단계 4: 의사결정 회고

JOURNEY.md의 결정 중 **다시 선택한다면 다르게 갈 수 있는 항목**을 짧게 짚어본다. 후회가 아니라 "다른 선택지가 어떤 트레이드오프였는지" 환기 목적이다.

- **CSI Secret Driver vs Sealed Secrets** (ch6.2): managed CSI는 GCP 종속, Sealed Secrets는 클라우드 중립이지만 키 회전 부담
- **Canary 도입 시점** (ch6.3): Blue/Green 직후가 아니라 멀티테넌시 이후로 미뤘으면 트래픽 분배 시연이 더 풍부했을지
- **Valkey vs Redis Streams** (ch6.1): 메시지 큐는 ch8 Kafka로 갔지만 Streams로 단일 컴포넌트로 묶는 선택지도 있었음

각 항목마다 "지금은 이렇게 갔지만 다음 프로젝트에서는 X도 검토 가치가 있다"로 마무리한다.
